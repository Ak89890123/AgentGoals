### MEDIUM

1. Global registry path contracts are documented but not enforced.

   The docs say `repo_root`, `goal_root`, and `state_path` are absolute/explicit paths (`docs/global-registry-schema-notes.md:15`, `docs/global-registry-schema-notes.md:16`, `docs/global-registry-schema-notes.md:17`, `docs/global-registry-schema-notes.md:27`), but the schema only requires non-empty strings (`schemas/global-registry.schema.json:42`, `schemas/global-registry.schema.json:46`, `schemas/global-registry.schema.json:50`), and runtime loading simply wraps the strings in `Path(...)` without checking `is_absolute()` (`src/goal_lifecycle/aggregate.py:56`, `src/goal_lifecycle/aggregate.py:57`, `src/goal_lifecycle/aggregate.py:58`). A relative `state_path` would be resolved against the operator's current working directory, which makes the read allowlist cwd-dependent on Windows. The tests only use resolved absolute paths (`tests/test_aggregate.py:23`, `tests/test_aggregate.py:24`, `tests/test_aggregate.py:25`) and do not cover rejection of relative paths.

2. Aggregated entries are copied without rebasing or checking path membership under the registered repo/goal root.

   `read_repo_state()` validates the repo STATE schema, then `entry_from_payload()` copies `goal_root`, `contract_path`, `plan_path`, and `evidence_path` directly from each repo-local STATE entry (`src/goal_lifecycle/aggregate.py:86`, `src/goal_lifecycle/aggregate.py:91`, `src/goal_lifecycle/aggregate.py:95`, `src/goal_lifecycle/aggregate.py:96`, `src/goal_lifecycle/aggregate.py:97`, `src/goal_lifecycle/aggregate.py:98`). The aggregator does not use `root.repo_root` to normalize relative repo-local paths and does not verify that entry paths stay under `root.goal_root`. In the current repo, the local registry uses a relative `goal_root` (`registry/REGISTRY.json:8`), so the generated global output contains relative/backslash paths. This does not mutate source files, but it weakens global dashboard accuracy and registry-scope semantics once multiple repos are aggregated.

### LOW

1. Repo STATE BOM handling is asymmetric with global registry BOM handling.

   Global registry loading explicitly uses `utf-8-sig` (`src/goal_lifecycle/aggregate.py:48`) and has a test for it (`tests/test_aggregate.py:108`), but repo STATE validation and reading go through `validate_state()`/`load_json()` using `utf-8` (`src/goal_lifecycle/validate.py:14`, `src/goal_lifecycle/validate.py:19`, `src/goal_lifecycle/aggregate.py:72`, `src/goal_lifecycle/aggregate.py:73`). That is fine for STATE generated by this repo, but a Windows-authored or externally generated repo STATE with a BOM will be reported as `parse_error`. If cross-repo STATE files may be hand-created or produced by other tools, add an explicit test and consistent decoding policy.

2. Test coverage does not include multiple registered repos or duplicate goal ids.

   The tests cover merge, missing STATE, malformed STATE, inactive roots, missing `state_path`, and registry BOM (`tests/test_aggregate.py:34`, `tests/test_aggregate.py:54`, `tests/test_aggregate.py:69`, `tests/test_aggregate.py:85`, `tests/test_aggregate.py:97`, `tests/test_aggregate.py:108`). They do not cover two active roots with overlapping goal ids/projects or path disambiguation. This is not a blocker for the single registered harness root, but it is a natural next test before relying on the global aggregate as a multi-repo operator dashboard.
